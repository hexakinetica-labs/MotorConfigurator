# ETHERCAT: Motor Type + EEPROM (формализация проблемы и план реализации)

Дата: 2026-03-19 04:26 (UTC+4)  
Статус: Draft for boss approval

---

## 1) Контекст задачи

Бизнес-задача: инженер должен
1) считать/зафиксировать конфиг с первой EtherCAT-оси,
2) подключить ещё 2 оси,
3) прошить их тем же конфигом.

Критичный параметр в этом процессе: **тип мотора (Motor Type / Motor selection)**, так как он определяет hardware-класс привода/мотора (мощность/модель).

---

## 2) Подтверждённые факты (документация)

### 2.1 Таблица кодов моторных моделей
Источник: `P100E EtherCAT AC Servo Drive User manual.docx.pdf`, стр. 20.

Подтверждены соответствия (пример ключевых):
- `0 -> 40-00130`
- `2 -> 60-00630`
- `3 -> 60-01330`
- `6 -> 80-02430` (default)
- `7 -> 80-03520`

### 2.2 Требование unlock перед изменением
Источник: тот же manual, стр. 23.

Подтверждено, что перед изменением `Motor selection` требуется установить пароль:
- `PA0 = 385`

### 2.3 Объекты EtherCAT
Источник: `DVS_ETHCAT_V1.12_XJ_3.xml` (ESI XML).

Подтверждено наличие:
- `0x2007:1` — `Motor Type`
- `0x2005:3` — `Is Param Loaded to EEPROM` (признак/механизм, связанный с загрузкой/сохранением параметров в EEPROM)

---

## 3) Формализация проблемы

На текущем этапе проект не предоставляет завершённый и безопасный пользовательский workflow:
- изменить `Motor Type`;
- сохранить изменение в долговременную память привода (EEPROM);
- верифицировать, что после перезапуска значение сохранилось.

Следствие: cloning flow для EtherCAT потенциально **неполный** для hardware-критичных параметров и может дать рассогласование между осями.

---

## 4) Риски, если оставить как есть

1. **Неполное клонирование**: часть параметров переносится, а `Motor Type` остаётся старым.  
2. **Опасная конфигурация**: неверный motor class для конкретной механики.  
3. **Ложное чувство готовности**: в runtime «похоже применилось», но после power cycle всё откатилось.  
4. **Сервисные потери времени**: ручные повторные настройки на каждой оси.

---

## 5) Предлагаемое решение (MVP, без overengineering)

Реализовать **один узкий use-case**:

`set_persistent(PersistentCommand::MotorType, code)`

Где внутри EtherCAT-адаптера выполняется последовательность:
1) unlock (`PA0=385`),  
2) запись `Motor Type` (`0x2007:1`),  
3) trigger сохранения в EEPROM (через подтверждённый path, связанный с `0x2005:3`/vendor flow),  
4) readback-проверка значения.

Это закрывает задачу «сделать и использовать сейчас», не требуя большого платформенного рефакторинга.

## Шаг 2. Добавить support `Motor Type` в dictionary

Файлы (ориентировочно):
- `drivers/interfaces/ethercat/p100e_ethercat_dictionary.h`

Что сделать:
- добавить/проверить descriptor для `0x2007:1`;
- пометить как hardware-критичный параметр в комментариях/метаданных (минимум — в документации к коду).

Definition of Done:
- параметр виден в списке EtherCAT-параметров и корректно читается.

---

## Шаг 3. Реализовать узкий API `set_persistent(...MotorType...)`

Файлы (ориентировочно):
- `motion_core/include/motion_core/axis_interface.h`
- `drivers/interfaces/ethercat/adapter/ethercat_axis_adapter.h`

Что сделать:
- добавить enum `PersistentCommand` (минимум: `MotorType`);
- добавить метод `set_persistent(PersistentCommand, int32_t)`;
- в `EthercatAxisAdapter` реализовать full-sequence: unlock -> write -> eeprom-save -> readback.

Definition of Done:
- метод возвращает success только при успешном прохождении всех этапов.

## Шаг 4. Добавить минимальный UI-flow для инженера

Файлы (ориентировочно):
- `ui/qt/mks/axis_manager.h/.cpp`
- `ui/qt/mks/axis_workspace.h/.cpp`

Что сделать:

- кнопка `Write Motor Type to EEPROM`;
- confirm-dialog с предупреждением (hardware-critical parameter);
- вывод статуса операции и readback.

Definition of Done:
- оператор может выполнить полный workflow из UI без ручных SDO-команд.

---

## Шаг 5. Валидация на реальном стенде

Сценарий:
1) прочитать исходный `Motor Type`,
2) записать новый code,
3) сохранить в EEPROM,
4) выполнить power cycle,
5) повторно прочитать и сверить.

Definition of Done:
- значение после power cycle соответствует записанному;
- операция стабильно повторяется на 3 осях.

---

## 7) Что сознательно НЕ делаем в MVP

- не строим общий framework persistent-параметров для всех устройств;
- не переделываем полностью архитектуру UI/CLI;
- не включаем mass-clone hardware-параметров «по умолчанию».

Это отдельные фазы после подтверждения MVP.

---

## 8) Критерии приёмки для согласования с руководителем

1. Есть рабочая операция `Motor Type -> EEPROM` с readback-верификацией.  
2. Операция доступна инженеру из UI понятным отдельным действием.  
3. После power cycle значение сохраняется.  
4. Документирован безопасный порядок применения для серийной наладки 3 осей.

---

## 9) Решение, которое требуется утвердить

Утвердить MVP-объём:
- реализуем только `PersistentCommand::MotorType`;
- UI-кнопка `Write Motor Type to EEPROM`;
- валидация на 3 осях;
- дальнейшее расширение в generic persistent API — отдельной задачей.
