

- __Бесконечный рост `rx_buffer_` (Уязвимость / Утечка)__: Если клиент или сервер начнут слать данные без символа `\n`, `rx_buffer_` будет расти до тех пор, пока не закончится оперативная память. Нет защиты от переполнения буфера (Max Buffer Size).

- __Небезопасная остановка драйвера__: В `HalHostDriver::stop()` мы ставим флаг `io_running_ = false` и делаем `io_thread_.join()`. Но если поток заблокирован в `recv()` (сеть отвалилась, но таймаут еще не вышел), драйвер зависнет при остановке/выключении робота.

__Несогласованность паттерна Producer-Consumer__: В `HalHostDriver` методы `write()` и `read()` просто обновляют/читают закешированные данные под мьютексами, в то время как `io_thread_loop` крутится сам по себе. Motion Pipeline обычно требует жесткой синхронизации: записали target -> дождались отправки -> получили actual -> отдали в контроллер. Можно ли сдесь сделать без блокироки?

__Обработка E-Stop (Аварийной остановки)__: Сброс E-Stop обрабатывается как обычный кадр в общем потоке (с JSON парсингом). Такие вещи должны обрабатываться по отдельному, высокоприоритетному каналу с минимальными задержками. - Estop должен дизейблит все оси 
- __Блокировка всего сервера (Deadlock-риск)__: В `force_disable_ethercat_axis` есть цикл с `std::this_thread::sleep_for(5ms)`. Эта функция вызывается из `apply_axis_op`, которая вызывается из `handle_control_frame`, где __удерживается глобальный `mutex_`__. В итоге отключение оси полностью замораживает весь HAL Host Service для всех клиентов на несколько десятков миллисекунд.

### 🕸 2. Бесполезный поток `io_thread_loop` в `HalHostDriver`

Драйвер создает отдельный поток `io_thread_loop`, который делает `sleep_for(4ms)`, просыпается, захватывает мьютекс `command_mutex_`, копирует кадр в `latest_command_frame_` и только потом отправляет. Вся эта конструкция с мьютексами, кешированием флагов (`estop_active_cache_`, `motion_epoch_cache_`) и отдельным потоком — __чистый оверинжиниринг__. __Решение:__ Раз уж `write()` вызывается конвейером (motion pipeline) с нужной частотой, то отправку данных можно делать прямо из метода `write()` (или писать в Lock-Free очередь), полностью удалив `io_thread_loop` и все связанные с ним мьютексы. Схлопнуть кеши!

### 🍔 3. JSON внутри JSON'a

В `HalHostService::apply_axis_op` для операций `EnqueueMotionBatch`, `ApplyParameterPatch` и `SetPersistent` происходит парсинг `service_string_value`: `const auto parsed = json::parse(frame.service_string_value, nullptr, false);` То есть, сначала мы распарсили внешний JSON кадра, достали оттуда строку, и теперь парсим её как еще один JSON! Это двойная десериализация, съедающая процессор впустую.
